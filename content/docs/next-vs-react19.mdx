# Next.js 15 与 React 19 的取舍分析

React 19 带来了编译期优化与 Actions API，但单独使用仍然需要自行处理路由、数据抓取和部署协议。Next.js 15 则把这些能力打包在 App Router 中，并继续强化服务器组件生态。

## 核心差异一览

| 维度 | Next.js 15 | React 19 + 自建栈 |
| ---- | ---------- | ---------------- |
| 数据获取 | 内置 Server Actions、缓存策略 | 需组合 React Router/TanStack Query |
| 路由系统 | 文件系统路由，支持并行/拦截路由 | 需要额外库实现 |
| 构建部署 | Vercel 原生支持，提供优化好的 RSC 和 Turbopack | 需手动配置 bundler 与服务器 |
| 文档与生态 | Fumadocs、shadcn/ui 等成熟模板 | 自由度高但维护成本更高 |

## 何时选择 Next.js

- 团队希望专注业务逻辑，不想重复造基础设施。
- 项目需要 SSR/SSG、边缘部署或图片优化等完整方案。
- 想结合 App Router + RSC 获取更好的性能与体验。

## 何时选择纯 React

- 产品对运行环境要求极为特殊，需要完全自定义打包与渲染管线。
- 已经有稳定的内部平台（例如微前端架构）提供路由和部署能力。
- 希望使用 React 19 的全套实验能力并自行控制升级节奏。

> 建议策略：以 Next.js 15 为默认首选，再根据项目复杂度和运维能力评估是否拆分为独立 React 19 栈。
